# D30 — Nested routes: children arrays, chain-prefix reuse, root-only layouts (v1.3)

A routed **view** can host a child routed view at its `<Slot/>` via a `children: [...]` array on the parent route; navigation reuses the shared chain prefix, layouts stay root-only, and params merge down the chain. A router-only, purely additive amendment. Settled (v1.3); see [[DOC-SPEC]] §9, [[DOC-ROUTER]], and [[DOC-VIEW-LIFECYCLE]] §4.

## Context
v1's router ([[DECISION-D19-NAVIGATION-COMMIT]]) is strictly flat: one route = one full-path regex → one view + optional top-level layout, and the only nesting is the layout's `<Slot/>` hosting the routed view. D30 lets a routed **view** host a child routed view the same way — a route object gains `children: [...]` and the parent view renders its matched child at `<Slot/>`. The slot/compose/destroy plumbing is already generic (the compiler parses `<Slot/>` in any template; keyed component vnodes with preattached instances compose at any depth; `destroy()` cascades to descendants), so this is a **router-only** amendment — no compiler, ViewManager, or PuzzleView changes. Like [[DECISION-D28-ANIMATIONS]]/[[DECISION-D29-LOOP-COUNTER]] it is purely additive: a route with no `children` (chain length 1) behaves exactly as today, and all v1 route contracts (trailing-slash sensitivity, `'*'` catch-all matched last, D19 commit ordering) are unchanged.

## Decision
Sub-decisions, each with its rejected alternative:

- **Nesting is declared by a `children: [...]` array on the parent route.** The `children` array **is** the chain, in declaration order, and the compiler flattens it to one compiled entry per leaf (`{ chain, fullPaths, regex, paramNames, layout }`). Matching stays depth-first in declaration order — the same first-match-wins contract as flat v1. (Rejected: a flat routes array with children referring to a parent by `name` — see Alternatives rejected.)
- **Child paths are relative to the parent; absolute child paths throw.** `{ path: '/settings', children: [{ path: 'profile' }] }` composes to `/settings/profile` (join rule: strip one trailing slash off the parent, add `/`, add the child). A leading `/` on a child path is a **constructor throw** (fail-fast config error), alongside `layout` on a non-root node, `path: '*'` inside `children`, and a duplicate `:param` name within one chain. (Rejected: absolute child paths repeating the parent prefix — see Alternatives rejected.)
- **`layout` is a root-only field; children inherit the chain's layout.** Layouts are **root gates/shells** — an auth wall, the app chrome — so they belong only at depth 0. The clean invariant: a layout swap can happen **only** when the chain diverges at depth 0 (`keep === 0`); at any deeper divergence the layout is necessarily reused. `layout` on a child node is a constructor throw. (Rejected: per-level layouts — see Alternatives rejected.)
- **A parent with children does NOT match its own bare URL; opt in with an explicit `path: ''` index child.** A bare parent URL with no index child **falls through to the catch-all** (`'*'`) like any unmatched URL. To match the bare URL, add a child with `path: ''` — it composes to exactly the parent path (`'/settings'` + `''` → `'/settings'`) and fills the slot. This makes "what shows at the parent URL" an explicit, visible decision. (Rejected: auto-matching the bare parent URL and rendering with an empty `<Slot/>` — see Alternatives rejected.)
- **Ancestor refresh is awaited pre-commit — the D19 gate, generalized over the chain.** On navigation, `keep` = the shared-prefix length of the old vs new `chain` (route-node identity). Fresh views (`[keep..N]`) `preload({params, props:{}})`; **reused** ancestor views (`[0..keep-1]`) `refresh({params})` with the full merged params — and **all** of these are `await`ed before the URL commits (`pushState` + title). Routed ancestor content gates the URL exactly as the leaf's `data()` does in flat v1: a rejection or a superseded nav token destroys the fresh instances only, leaves reused ancestors in place, and changes neither URL nor title. This deletes the old separate params-only branch: `cur.entry === entry` is just `keep === chain length`, the degenerate whole-chain-refresh case. **Accepted soft-violation (unchanged from D19):** if an ancestor's refresh resolves before a deeper sibling's load *rejects*, that ancestor has already re-rendered under the old URL — the same hazard the flat params-only failure path already carried, not a new one. (A reused **layout**'s refresh stays *post*-commit — it is chrome, not routed content — preserving the existing layout-reuse tests.) (Rejected: committing the URL before ancestor `data()` settled — see Alternatives rejected.)
- **One animator per transition generalizes to "the topmost swapped instance animates; everything below rides along."** The D28 rule (reused layout → view animates alone; layout swap → layout animates as the unit) is the two-level case of this. In a chain, the animator is `views[keep]` — the shallowest instance that actually changed — and every fresh instance below it is `skipEnter()`'d (each nested `mountComponent` auto-chains a `playIn()` that must be suppressed, or the whole subtree would animate at once). Consequences: `keep === 0` is the layout-swap / whole-chain case (animator = `cur.layout ?? cur.views[0]`); `keep ≥ 1` is the new mid-chain case (layout reused, `views[keep]` animates, deeper fresh views `skipEnter()`). (Rejected: animating each swapped level independently — see Alternatives rejected.)
- **After the out, the router hands the FULL rebuilt vnode chain to the topmost host** (the layout, or the surviving root view when there is none) and lets the keyed patch cascade down to the divergence level. The cascade also makes the reused layout's post-commit `data()` re-run safe at any depth, so the layout refresh behaves uniformly. (Rejected: swapping only at the survivor via `views[keep-1].applyParentUpdate` with just the fresh sub-chain — see Alternatives rejected.)
- **Interruption: the pending-out unit and everything under it is DOOMED, and the winner must not build on it.** A navigation that arrives while a previous transition is still animating out clamps its reuse prefix to exclude that subtree (`#pendingOut === cur.layout` → nothing survives, fresh layout; `#pendingOut === cur.views[j]` → `keep ≤ j`) — without the clamp, a winner that "reuses" an instance mid-teardown commits a blank subtree (reproduced; regression-tested). Conversely, when the pending-out is *deeper* than the winner's own old-animator, destroying `#pendingOut` does not cover it, so a different still-live old-animator is destroyed synchronously too (skip its out — we are interrupting).
- **Vnode keys: a reused level keeps its committed key; a fresh level gets its accumulated path *pattern* (`fullPaths[i]`) stamped with the navigation token — never runtime params.** `patchComponent()` reuses the old instance whenever tag+key match and ignores a preattached `vnode.instance`, so the key must change exactly when the level's *instance* changes. The generation stamp (`fullPaths[i] + '\x00' + navToken`) makes every fresh level's key unique per navigation while reused levels carry their committed key forward, so the keyed patch reuses exactly the levels the router reused and swaps exactly the levels it rebuilt. (Rejected: keying on the runtime-substituted path, and the bare compile-time pattern alone — see Alternatives rejected.)
- **`meta.title` resolves nearest-defined, walking leaf → root.** The committed `document.title` is the first `meta.title` found scanning from the matched leaf up its ancestors. A leaf may omit `title` and inherit a section title from its parent; a leaf that defines one overrides. (Rejected: requiring `title` on every leaf, or taking only the root's — see Alternatives rejected.)

## Alternatives rejected
- **A flat routes array with children referring to a parent by `name`** (`{ path: '/settings/profile', parent: 'settings' }`): scatters one visual hierarchy across unordered top-level entries, forces a name-resolution pass with its own missing-parent/cycle errors, and hides the chain the runtime actually walks.
- **Absolute child paths** (leading `/`, repeating the parent prefix): duplicates the prefix at every level, so renaming a parent segment silently desyncs its children, and it invites a child path that doesn't actually sit under its parent.
- **Per-level layouts** (a `layout` on any node, each wrapping its subtree): produce layout-inside-view sandwiches (a layout nested inside a routed view inside another layout), multiply the animation cases (every depth could swap a layout), and confuse what a layout *is*.
- **Auto-matching the bare parent URL and rendering the parent with an empty `<Slot/>`:** an empty slot hole is almost always a mistake (the author forgot the index pane), and silently rendering a gap hides it.
- **Committing the URL before ancestor `data()` settled:** reintroduces the D19 desync bug one level up.
- **Animating each swapped level independently:** compounding durations and incoherent double motion — the same reason [[DECISION-D28-ANIMATIONS]] rejected animating layout-and-view together.
- **Swapping only at the survivor** (`views[keep-1].applyParentUpdate` with just the fresh sub-chain): every host *above* the survivor would keep vnodes whose descendants describe the old chain, and `patchComponent` pushes slot children through on every re-render — so any later ancestor re-render (a store change, a `setData`) would push that stale content back down, destroying the fresh subtree and resurrecting destroyed instances (reproduced; regression-tested).
- **Keying on the runtime-substituted path** (e.g. `/user/42`): two navigations to different params of the *same* route would collide on the key and adopt the wrong (stale) instance, and — worse — two *sibling children sharing one view class* would fail to swap.
- **The bare compile-time pattern alone as key:** after an interruption clamp, the *same* pattern can legitimately re-enter with a fresh instance while the old, destroyed one still sits in the host's tree under that key — a bare-pattern key would match and adopt the corpse.
- **Requiring `title` on every leaf (or taking only the root's):** leaf-first-with-fallback matches how nested UIs actually title themselves (specific page name when it has one, section name otherwise).

## Consequences
**Params:** the full regex of the leaf's `fullPaths[N]` yields **one merged params object**, and **every level's `data(params)` receives the full merged params** (matching how the layout already receives the routed view's params in flat v1). A duplicate `:param` name within one chain is a constructor throw (it would make the merged object ambiguous).

**Documented constraint:** a parent whose template lacks a `<Slot/>` preloads its child but never mounts it (nothing hosts it) — a dev warning fires post-commit when a non-leaf level's element is null. This is the nested analogue of the flat "layout must contain `<Slot/>`" requirement.

Non-breaking: flat routes are unchanged; this is an additive amendment (v1.3).
